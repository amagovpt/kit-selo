Checklist "10 aspetos críticos de acessibilidade funcional para sítios Web"

Nota: Para recolha de evidências relativa a esta Checklist utilize o ficheiro excel: Recolha de evidências da Checklist “10 aspetos críticos de acessibilidade funcional” (xlsx, 136KB)

A checklist "10 aspetos críticos de acessibilidade funcional" é a lista de verificação a que se faz alusão no artigo 9.º, n.º 1, alínea b) do Decreto-Lei n.º 83/2018 e deve ser usada de acordo com a metodologia referenciada no diploma:

  • "1. Para os sítios Web, as entidades referidas no artigo 2.º devem adotar os seguintes procedimentos de monitorização:
    • (...)
    • b) Procedimento simplificado manual, correspondente a uma avaliação manual pericial a uma amostra de páginas que permita responder à diversidade de elementos constantes da lista de verificação para sítios Web publicada no sítio Web www.acessibilidade.gov.pt;"

Nesta lista estão os 10 aspetos críticos de acessibilidade funcional para sítios Web encontrados nos diversos estudos aos sítios Web da Administração Pública Portuguesa elaborados pela equipa da Unidade ACESSO - que integra atualmente a Equipa de Experiência Digital da AMA - durante os últimos 20 anos. Esta lista baseia-se também na lista de verificação do W3C Easy Checks - A First Review of Web Accessibility.

Requisitos a cumprir:

1 - Menus de Navegação

1.1 O menu de navegação deve estar estruturado como uma lista de opções

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 1.3.1 (nível A) - Informações e Relações
  • 2.4.1 (nível A) - Contornar Blocos
  • 3.2.3 (nível AA) - Navegação Consistente

Técnicas WCAG:

  • H48: Utilizar ol, ul e dl para listas ou grupos de links
  • H97: Agrupar links relacionados utilizando o elemento nav
  • SCR28: Utilizar um menu expansível e colapsável para contornar blocos de conteúdo

Falhas WCAG:

  • F43: Falha do Critério de Sucesso 1.3.1 derivado da utilização de uma marcação estrutural que não reflete as relações estabelecidas no conteúdo

Para que possa ser bem interpretado por tecnologias de apoio, os menus e submenus devem estar estruturados com elementos nativos, do tipo <ul>, ou com a semântica e o estado dos elementos identificados com técnicas em ARIA.

1.2 É possível selecionar as opções e as subopções do menu quer com rato quer com teclado

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 2.1.1 (nível A) - Teclado
  • 2.4.3 (nível A) - Ordem do Focus
  • 2.4.7 (nível AA) - Focus Visível

Técnicas WCAG:

  • G202: Garantir que toda a funcionalidade é controlada pelo teclado
  • G90: Fornecer manipuladores de eventos acionados por teclado
  • C27: Fazer com que a ordem DOM corresponda à ordem visual
  • G149: Utilizar componentes de interface de utilizador que ao receber o focus são destacados pelo agente de utilizador
  • C15: Utilizar CSS para alterar a apresentação de um componente da interface de utilizador quando o mesmo recebe o focus
  • G165: Utilizar por defeito o indicador de focus da plataforma permite aceder por defeito aos indicadores de focus de elevada visibilidade
  • G195: Utilizar um indicador de focus visível fornecido pelo autor
  • C40: Criar um indicador de focus de duas cores que garanta contraste suficiente com todos os componentes
  • C45: Utilizar :focus-visible de CSS para fornecer indicação de focus do teclado
  • SCR31: Utilizar programação do lado do cliente para alterar a cor de fundo ou a cor da extremidade do elemento com focus

Falhas WCAG:

  • F54: Falha do Critério de Sucesso 2.1.1 devido à utilização para uma função apenas de manipuladores de eventos específicos para dispositivos apontadores (incluindo gestos)
  • F42: Falha do Critério de Sucesso 1.3.1, 2.1.1, 2.1.3, ou 4.1.2 ao emular links
  • F44: Falha do Critério de Sucesso 2.4.3 devido à utilização de tabindex para criar uma ordem de tabulação que não preserve o significado e a operacionalidade
  • F85: Falha no Critério de Sucesso 2.4.3 devido à utilização de caixas de diálogo ou menus não adjacentes ao seu elemento acionador
  • F78: Falha do Critério de Sucess 1.4.11, 2.4.7 e 2.4.13 devido a estilizar extremidades e contornos do elemento que remove ou torna invisível o indicador visual do focus

Deve ser possível percorrer a estrutura de navegação quer com um dispositivo apontador quer com o teclado.

1.3 As imagens-link, caso existam no menu, devem ter o correspondente equivalente alternativo em texto

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 1.1.1 (nível A) - Conteúdo Não-Texto
  • 2.4.4 (nível A) - Propósito do Link (Em Contexto)

Técnicas WCAG:

  • G94: Fornecer alternativa em texto curto para conteúdo não-texto que serve o mesmo propósito e apresente a mesma informação que o conteúdo não-texto
  • G82: Fornecer uma alternativa em texto que identifique o propósito do conteúdo não-texto

Falhas WCAG:

  • F3: Falha do Critério de Sucesso 1.1.1 devido à utilização de CSS para incluir imagens que transmitem informação importante
  • F30: Falha do Critério de Sucesso 1.1.1 e 1.2.1 devido à utilização de alternativas em texto que não são alternativas (e.g., nomes de ficheiros ou texto placeholder)
  • F65: Falha do Critério de Sucesso 1.1.1 devido à omissão do atributo alt ou alternativa em texto nos elementos img, elementos area, e elementos input do tipo "image"
  • F71: Falha do Critério de Sucesso 1.1.1 devido à utilização de algo que se assemelha a texto para representar texto sem fornecer uma alternativa em texto
  • F89: Falha do Critério de Sucesso 2.4.4, 2.4.9 e 4.1.2 devido a não fornecer um nome acessível para uma imagem que é o único conteúdo do link

As imagens corretamente legendadas permitem ser interpretadas como texto, tornando todas as opções de navegação acessíveis.

2 - Títulos e Subtítulos

2.1 Existe um título <h1> marcado na página

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 1.3.1 (nível A) - Informações e Relações
  • 2.4.1 (nível A) - Contornar Blocos
  • 2.4.6 (nível AA) - Cabeçalhos e Etiquetas

Técnicas WCAG:

  • H42: Utilizar h1-h6 para identificar títulos
  • G130: Fornecer títulos descritivos
  • G115: Utilizar elementos semânticos para marcação da estrutura
  • G141: Organizar a página utilizando títulos
  • ARIA12: Utilizar role="heading" para identificar títulos
  • H69: Fornecer elementos de título no início de cada secção do conteúdo
  • T3: Utilizar convenções padronizadas de formatação de texto para títulos

Falhas WCAG:

  • F43: Falha do Critério de Sucesso 1.3.1 devido à utilização de uma marcação estrutural que não reflete as relações estabelecidas no conteúdo

O título principal de cada página, que sumariza o seu conteúdo, deve ser identificado como o primeiro nível dos títulos (h1). Não deverá ser utilizado mais do que um <h1> por página.

2.2 Existe uma marcação hierarquizada de títulos e subtítulos na página (<h1>...<h6>)

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 1.3.1 (nível A) - Informações e Relações
  • 2.4.1 (nível A) - Contornar Blocos
  • 2.4.6 (nível AA) - Cabeçalhos e Etiquetas

Técnicas WCAG:

  • H69: Fornecer elementos de título no início de cada secção do conteúdo
  • G130: Fornecer títulos descritivos
  • G115: Utilizar elementos semânticos para marcação da estrutura
  • G141: Organizar a página utilizando títulos
  • ARIA12
  • T3: Utilizar convenções padronizadas de formatação de texto para títulos<

Falhas WCAG:

  • F43: Falha do Critério de Sucesso 1.3.1 derivado da utilização de uma marcação estrutural que não reflete as relações estabelecidas no conteúdo

O título principal de cada página, que sumariza o seu conteúdo, deve ser identificado como o primeiro nível dos títulos (h1). Não deverá ser utilizado mais do que um <h1> por página.

Os títulos são empregues de forma hierárquica para melhor estruturar os conteúdos, das informações mais gerais às mais particulares. Deverão ser usados de forma consistente por todo o sítio Web.

6 - Contraste

6.1 No corpo de um documento, o rácio de contraste entre a cor do texto normal (menor que 18 pontos ou menor que 14 pontos negrito) e a cor do fundo é superior a 4,5:1

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 1.4.3 (nível AA) - Contraste (Mínimo)

Técnicas WCAG:

  • G18
  • G148
  • G174

Falhas WCAG:

  • F24
  • F83

Deve assegurar-se no corpo do documento que o rácio de contraste entre a cor do texto e a cor de fundo é, no mínimo, de 4,5:1, de forma a assegurar a sua legibilidade para utilizadores com deficiências da visão.

6.2 O rácio de contraste entre a cor do texto de tamanho grande (maior ou igual que 18 pontos ou maior ou igual que 14 pontos negrito) e a cor do fundo é superior a 3:1

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 1.4.3 (nível AA) - Contraste (Mínimo)

Técnicas WCAG:

  • G145

  • G174

Falhas WCAG:

  • F24
  • F83

7.2 O vídeo ou o áudio deve conter preferencialmente legendas fechadas sincronizadas. Caso não seja possível, no mínimo, deve disponibilizar-se uma transcrição textual

Relação do requisito com as WCAG 2.1

O uso de legendas fechadas destina-se essencialmente a pessoas surdas. Recomendam-se para a produção das referidas legendas técnicas de tradaptação conhecidas para o efeito bem como o enriquecimento das legendas de sons cuja mensagem não seja percetível visualmente (por ex., o toque de uma campaínha de uma porta).

A audiodescrição é fundamental para que pessoas cegas ou com baixa visão possam percecionar a mensagem veiculada.

9 - Modais

9.1 Quando a caixa de diálogo é aberta, o foco (cursor do Browser) move-se para um elemento dentro da caixa de diálogo

Relação do requisito com as WCAG 2.1

Nota: o presente requisito baseia-se no padrão de construção de caixas de diálogo (tipo Modal) do W3C/WAI.

Critérios de Sucesso WCAG:

  • 1.3.1 (nível A) - Informações e Relações
  • 2.4.3 (nível A) - Ordem do Focus

Técnicas WCAG:

  • SCR26: Inserir de imediato conteúdo dinâmico no Document Object Model (DOM) assim que o seu controlo acionador é acionado
  • H102: Criar caixas de diálogo (modais) com o elemento HTML dialog

Falhas WCAG:

  • F85: Falha no Critério de Sucesso 2.4.3 devido ao uso de caixas de diálogo ou menus não adjacentes ao seu elemento acionador

Assim que a caixa de edição se sobrepõe à janela, o cursor do Browser deve saltar para dentro da caixa de diálogo.

9.2 Quando uma caixa de diálogo está aberta, a navegação com teclado (Browser ou Tecnologia de apoio) tem de ficar circunscrita aos elementos que compõem a caixa de diálogo

Relação do requisito com as WCAG 2.1

Nota: o presente requisito baseia-se no padrão de construção de caixas de diálogo (tipo Modal) do W3C/WAI.

Critérios de Sucesso WCAG:

  • 1.3.1 (nível A) - Informações e Relações
  • 2.4.3 (nível A) - Ordem do Focus

Técnicas WCAG:

  • SCR26: Inserir de imediato conteúdo dinâmico no Document Object Model (DOM) assim que o seu controlo acionador é acionado
  • H102: Criar caixas de diálogo (modais) com o elemento HTML dialog

Falhas WCAG:

  • F85: Falha no Critério de Sucesso 2.4.3 devido ao uso de caixas de diálogo ou menus não adjacentes ao seu elemento acionador

Quando a caixa de diálogo está aberta, todos os outros elementos que se encontram fora da caixa devem permanecer inertes. Mesmo os elementos interativos (links, botões ou campos de edição) devem deixar de ser focáveis quer com teclado quer com o rato. Quando a caixa de diálogo está aberta, se ativar um leitor de ecrã e ele efetuar a leitura ou permitir focar elementos interativos que estão fora da caixa de diálogo, algo está errado.

9.3 A caixa de diálogo tem de ter um mecanismo que permita sair ou fechar a caixa, quer através de teclado quer através de um dispositivo apontador

Relação do requisito com as WCAG 2.1

Nota: o presente requisito baseia-se no padrão de construção de caixas de diálogo (tipo Modal) do W3C/WAI.

Critérios de Sucesso WCAG:

  • 1.3.1 (nível A) - Informações e Relações
  • 2.4.3 (nível A) - Ordem do Focus

Técnicas WCAG:

  • SCR26: Inserir de imediato conteúdo dinâmico no Document Object Model (DOM) assim que o seu controlo acionador é acionado
  • H102: Criar caixas de diálogo (modais) com o elemento HTML dialog

Falhas WCAG:

  • F85: Falha no Critério de Sucesso 2.4.3 devido ao uso de caixas de diálogo ou menus não adjacentes ao seu elemento acionador

A caixa de diálogo deve ter um mecanismo que permita sair ou fechar a caixa. Pode ser um botão "Fechar". Para além do botão "Fechar" é possível adicionar a tecla de atalho "ESC".

9.4 Quando a caixa de diálogo fecha, o foco (cursor do Browser) deve voltar ao elemento interativo que o invocou

Relação do requisito com as WCAG 2.1

Nota: o presente requisito baseia-se no padrão de construção de caixas de diálogo (tipo Modal) do W3C/WAI.

Critérios de Sucesso WCAG:

  • 1.3.1 (nível A) - Informações e Relações
  • 2.4.3 (nível A) - Ordem do Focus

Técnicas WCAG:

  • SCR26: Inserir de imediato conteúdo dinâmico no Document Object Model (DOM) assim que o seu controlo acionador é acionado
  • H102: Criar caixas de diálogo (modais) com o elemento HTML dialog

Falhas WCAG:

  • F85: Falha no Critério de Sucesso 2.4.3 devido ao uso de caixas de diálogo ou menus não adjacentes ao seu elemento acionador

É útil que ao sair da modal, o utilizador seja posicionado no elemento que usou para invocar a modal. A modal é uma espécie de parêntesis à conversa principal. É bom que o utilizador, depois do "parêntesis", possa voltar à conversa principal, no exato ponto onde a deixou.

10 - Ficheiros PDF

10.1 Nos ficheiros PDF é possível, no mínimo, extrair o conteúdo textual para formato TXT

Relação do requisito com as WCAG 2.1

Critérios de Sucesso WCAG:

  • 1.4.5 (nível AA) - Imagens de texto

Técnicas WCAG:

  • PDF7: Passar o documento PDF digitalizado por um software de OCR (Reconhecimento Óptico de Carateres) para permitir o acesso ao texto real

Os ficheiros PDF devem ter o seu texto inteiramente extraível para que se possa passar o respetivo conteúdo para um processador de texto sem perda de informação.
